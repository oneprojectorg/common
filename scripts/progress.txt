## Codebase Patterns
- Navigation config is in `apps/app/src/components/decisions/ProcessBuilder/navigationConfig.ts` with `as const` patterns for type derivation
- Content registry maps step/section â†’ React components in `contentRegistry.tsx`
- Translation keys use the string itself as the key (e.g., `'Overview'`, `'Phases'`)
- Uses `nuqs` for URL-based query state management
- Section IDs: overview, phases, proposalCategories, templateEditor, criteria, roles, participants
- Step IDs: general, template, rubric, participants
- `TranslationKey` type comes from `@/lib/i18n`
- Instance phases use `phaseId` field; template phases (PhaseDefinition) use `id` â€” normalize when mapping
- `trpc.decision.getInstance` query is shared across useNavigationConfig, PhasesSectionContent, and sidebar â€” all hit the same cache
- Footer re-implements Launch/Save logic from the old header: `trpc.decision.updateDecisionInstance` mutation for saves, `LaunchProcessModal` for draft launches
- `ProcessStatus` enum from `@op/api/encoders` for draft/published/completed/cancelled status checks
- `useProcessBuilderValidation(decisionProfileId)` returns `ValidationSummary` with `stepsRemaining`, `isReadyToLaunch`, and `checklist`
- useProcessNavigation now uses a single `section` query param (flat model) with backward compat `step` param cleanup
- visibleSections is derived from SIDEBAR_ITEMS filtered by navigationConfig (steps enabled + sections allowed)
- currentStep is derived from currentSection's parentStepId for backward compat with header/sidebar consumers
- ProcessBuilderContent uses getContentComponentFlat for flat sectionâ†’component lookup
